# LazyShell

AI Enhanced Server Management Terminal，用自然语言管理远程服务器。

## 项目简介

LazyShell 是一个基于 Tauri + React 的桌面终端应用，目标是把自然语言、安全确认和 SSH 运维能力组合到一个本地客户端里。用户可以通过 AI 将自然语言转换为 Linux 命令，在执行前查看命令内容，并通过 SSH 连接远程服务器完成操作。

它不是一个 Web 管理面板，而是一个偏“本地运维工作台”的产品：

- AI 负责理解意图并生成命令
- Rust 后端负责 SSH、加密和本地持久化
- React 前端负责多服务器、多标签、终端与 AI 协作界面

当前项目已从 Claude Code 阶段转入 Codex 继续开发，本文档作为统一的项目说明与开发入口，后续以 `README.md` 为主，不再让说明分散在多份文档里。

## 核心能力

- 自然语言转 Linux 命令
- 多 AI Provider 支持，运行时切换
- SSH 远程连接与命令执行
- 多标签页服务器管理
- 交互式终端体验
- 服务器配置本地加密存储
- 危险命令识别与二次确认
- 命令学习与会话记忆

## 技术栈

- 前端: React 19 + TypeScript + Vite
- 后端: Rust + Tauri 2.x
- 终端: xterm.js + xterm-addon-fit + xterm-addon-web-links
- SSH: `ssh2`
- 加密: AES-256-GCM + PBKDF2
- AI: MiniMax / OpenAI / Anthropic

## 用户操作路径分析

下面按真实使用流程梳理当前已经完成的用户旅程。

### 路径 1: 首次打开应用

1. 用户进入解锁页。
2. 若本地不存在主密码，则进入设置模式。
3. 设置主密码后应用解锁，进入主界面。
4. 后续服务器配置和敏感信息通过该密码派生的密钥进行本地保护。

这一段说明“首次使用”已经不是裸奔状态，而是从一开始就把安全流程纳入入口。

### 路径 2: 添加并验证服务器

1. 用户在服务器侧边栏打开添加表单。
2. 输入名称、主机、端口、用户名和认证方式。
3. 选择密码或私钥认证。
4. 保存后后端进行配置落盘。
5. 用户可立即发起连接测试，验证配置是否有效。

这条路径已经形成完整的配置管理闭环。

### 路径 3: 打开一个服务器会话

1. 用户在侧边栏选择某台服务器。
2. 前端先请求创建 shell session。
3. 仅当后端 shell 创建成功后，才把该服务器加入 tab。
4. 主终端接管这个 `tabId` 的输入输出。
5. 同时异步获取 welcome banner。

这条路径设计得比较稳，避免了“tab 已出现但后端 shell 不存在”的孤儿状态。

### 路径 4: 在终端中执行交互操作

1. 用户输入内容进入 xterm。
2. 输入通过 `shell_input` 发给后端持久 shell。
3. 前端通过顺序轮询持续读取 `shell_output`。
4. 输出写回 xterm，形成交互式终端体验。
5. resize 时同步行列数，尽量保证全屏程序行为正常。

这条路径说明主终端已经不是伪终端，而是围绕持久 shell 设计的。

### 路径 5: 使用 AI 生成命令

1. 用户在 AI 面板输入自然语言需求。
2. 前端优先结合命令数据库、学习数据和记忆信息构建上下文。
3. AI 返回单条命令、多个候选，或要求澄清。
4. 用户确认后将命令发送给执行链路。
5. 后续结果可继续反馈到聊天、学习数据和命令卡片。

这条路径已经构成“提需求 -> 选命令 -> 执行 -> 沉淀经验”的主产品体验。

### 路径 6: 命令历史复用

1. 用户通过 AI、AI 历史命令、常用命令、内置命令或主终端手动输入触发命令。
2. 命令会根据入口附带来源标记。
3. 历史按服务器维度持久化到本地。
4. 用户后续可以在 AI 面板中直接搜索、筛选并重跑历史命令。

这条路径说明项目已经开始具备“跨会话工作台”特征，而不是一次性问答型 AI 工具。

## 架构协作分析

这一节关注前端、后端和本地存储目前如何协作。

### 前端职责

前端主要承担 4 类职责：

- 界面组织：侧边栏、tab、终端、AI 面板、设置弹窗
- 应用状态：活动服务器、tab、布局、命令历史、输出映射
- 用户输入采集：表单、终端输入、AI 输入
- 通过 Tauri `invoke` 调用后端能力

当前前端的核心协调点是 `src/App.tsx`，它相当于应用层控制器。

### 后端职责

后端主要承担 5 类职责：

- SSH 连接与命令执行
- PersistentShell 与 PTY 通道管理
- 服务器配置加密存储与解锁
- AI 请求代理与统一响应解析
- 记忆文件、学习数据和命令库读写

也就是说，所有真正敏感或系统级能力都在 Rust 端。

### 数据流协作

当前主要有 4 条关键数据流：

- 解锁流: UnlockScreen -> commands.rs -> crypto/storage
- 服务器流: ServerList -> commands.rs -> ssh manager + encrypted storage
- 终端流: Terminal -> shell_input/shell_output -> PersistentShell
- AI 流: AIChat -> aiProvider -> ai.rs -> external provider -> AIChat

这四条流已经构成项目主干。

### 当前架构优点

- 前后端分工基本合理
- 敏感能力留在 Rust
- 主终端与 AI 能力已能组合工作
- 本地记忆系统已经开始反哺 AI 上下文
- 命令历史已经形成“执行 -> 持久化 -> 复用 -> 反哺 AI”的闭环

### 当前架构主要问题

- `App.tsx` 承担过多协调职责，后续会继续膨胀
- 存在 `Terminal.tsx`、`PtyTerminal.tsx`、`InteractiveTerminal.tsx` 多条终端路线并存
- 类型系统和前后端契约尚未完全收口
- learning data 与 memory system 有功能重叠，需要逐步明确边界
- 当前命令历史仍是轻量采集，不是完整 shell transcript，交互式命令输出尚未精确关联

## 终端主线路分析

这一节专门回答“当前到底应该围绕哪套终端继续开发”。

### 当前三套终端的真实状态

#### `Terminal.tsx`

这是当前实际接入主界面的终端实现，也是当前唯一在线主路径。

它的链路是：

- `App.tsx`
- `Terminal.tsx`
- `shell_input` / `shell_output` / `shell_resize`
- `PersistentShell`

这条线路的特点是：

- 使用 xterm.js
- 按 tab 维度管理 shell 会话
- 已接入当前主布局
- AI 面板直接通过 `shell_input` 向它发命令
- 已处理 resize 和转义序列读取问题

这说明它不是候选方案，而是事实上的生产主线。

#### `PtyTerminal.tsx`

这是一套单独的 PTY 终端方案，但当前没有被 `App.tsx` 使用。

它的链路是：

- `PtyTerminal.tsx`
- `start_pty_session` / `pty_input` / `pty_output` / `pty_resize`
- `SSHConnection` 的 interactive channel

它的问题不是“不能用”，而是：

- 生命周期按 `serverId` 管理，不是按 tab 管理
- 输出轮询仍然是 `setInterval`
- 尺寸计算依赖手动字符宽高，而不是 xterm fit 后的真实 rows/cols
- 与当前主界面的 tab、AI、session 生命周期没有统一

所以它更像保留下来的备用路线或旧实验，而不是现在应该继续扩展的主线。

#### `InteractiveTerminal.tsx`

这不是一个真正的远程 PTY 终端，而是前端模拟的命令行交互视图。

它的特点是：

- 自己维护输入缓冲、光标位置和前端历史
- 依赖 `onCommandSubmit` 回调触发命令执行
- 输出显示基于 `history/currentOutput` 渲染
- 更像一个“命令控制台 UI”，不是完整终端协议实现

它适合做原型验证或简化交互界面，但不适合作为当前项目的长期主终端。

### 结论

当前应该明确：

- 保留 `Terminal.tsx + PersistentShell` 作为唯一主终端路线
- `PtyTerminal.tsx` 暂时降级为实验代码或迁移参考
- `InteractiveTerminal.tsx` 视为历史 UI 方案，不再作为主线继续扩展

### 为什么应该这样收敛

原因有四个：

1. 它已经接在当前 `App.tsx` 上，迁移成本最低。
2. 它按 `tabId` 管理 shell，会话模型更符合当前产品形态。
3. AI、布局、多 tab 这些上层功能已经默认依赖 `shell_input` 这套接口。
4. 你已经在这条线上修过转义序列和轮询竞态问题，经验已经沉淀在代码里。

### 建议的迁移原则

后续终端相关开发统一遵守这几个原则：

1. 所有主终端输入输出都走 `shell_*` 接口。
2. 会话生命周期统一以 `tabId` 为主键，而不是 `serverId`。
3. 行列同步统一以 xterm 的真实 `rows/cols` 为准，不再自己估算字符尺寸。
4. 输出读取统一使用顺序轮询，不再新增 `setInterval` 型并发轮询。
5. 命令历史优先通过统一入口写入，避免 AI、终端、侧边栏各自分叉记录。

### 建议的代码收敛顺序

#### 第一步

在文档和注释层面正式声明 `Terminal.tsx` 为主终端实现。

#### 第二步

把 `PtyTerminal.tsx` 标记为 legacy / experimental，避免后续误用。

#### 第三步

梳理哪些交互能力还只存在于 `PtyTerminal.tsx`，例如：

- 某些快捷键处理
- 右键粘贴
- 选中自动复制

有价值的能力迁回 `Terminal.tsx`，而不是继续双线维护。

#### 第四步

确认 `InteractiveTerminal.tsx` 是否还需要保留：

- 如果只用于历史参考，可以保留但不继续扩展
- 如果没有明确用途，可以后续移出主仓库实现目录

### 第二阶段: 统一终端与会话模型

这一阶段已完成主方向判断，并完成了主线增强：

1. 已明确 `Terminal.tsx` 为主实现
2. 备用终端路线已降级为 legacy / experimental
3. 主终端已补上基础交互增强，并将长时间后台切回、长时间前台空闲、连接类错误统一收口到“手动重连提示”
4. 新增终端相关功能优先走 `shell_*` 主路径
5. 已修复多 tab 切换时误清理 shell session、后台 tab 干扰前台会话导致的断连、卡顿与欢迎信息重复问题
6. 已修复终端舞台层叠命中问题，避免后开的 tab 覆盖先开的 tab 导致“看起来像卡死”
7. 重连动作现在会先清屏再重建会话，减少旧失效画面和新会话内容混杂

理由：

- 这是整个产品体验的地基
- 终端路线长期并存会拖累所有上层能力，包括 AI、历史、状态同步

### 第三阶段: 补齐持久化工作台能力

这一阶段已开始推进，当前状态如下：

1. 已完成按服务器维度的命令执行结果持久化
2. 已完成跨会话命令历史的基本复用
3. 已支持 AI、终端手动输入、历史重跑、常用命令、内置命令的来源标记
4. 已支持历史去重、搜索、来源筛选、裁剪和清空
5. 下一步重点应放在会话恢复、命令输出搜索与回放

理由：

- 这些能力会把产品从“会话工具”推向“工作台工具”
- 它们和已经存在的 memory system 天然兼容

### 服务器状态采集解耦

左侧服务器状态现在不再复用主终端相关的 SSH 会话，也不再依赖 `execute_command + session_pool` 这条共享链路。

当前实现改为：

1. 通过独立的 `get_server_status` 后端命令获取状态
2. 每次采集使用一条独立短连接
3. 在同一条短连接里顺序执行磁盘、内存、网络状态采样
4. 采集完成后立即断开

这样做的原因是：

- 服务器状态面板属于只读采集，不应该和主终端的持久 shell 争用 SSH 资源
- 主终端负责交互式 shell，会话应尽量稳定保留
- 左侧状态刷新应当对终端输入输出透明

这次拆分之后，终端交互和状态同步的职责边界更清楚，也更符合长期可维护的架构。

## 重要架构说明

### 1. 终端实现分层

当前主流程使用 `Terminal.tsx` 对接后端 `PersistentShell`。仓库中还保留了 `PtyTerminal.tsx` 与 PTY 相关接口，但它不是当前 `App.tsx` 的主线路。

可以简单理解为：

- `Terminal.tsx` 是当前主终端实现
- `PtyTerminal.tsx` 是备用 / 试验性 PTY 路线
- `InteractiveTerminal.tsx` 是另一套交互终端实现

后续开发需要优先统一这几条终端线路，避免长期并存。

### 2. 终端轮询约束

主终端读取输出时使用顺序轮询，而不是并发定时器轮询。这个约束很关键，尤其在 `vim`、`nano` 等全屏交互程序中，如果改成并发轮询，容易出现转义序列乱序和界面异常。

### 3. AI 上下文注入

AI 请求当前会注入这些上下文：

- 当前工作目录
- 最近命令及其输出摘要
- 当前服务器会话状态
- 部分本地记忆内容，例如常用命令和近期对话摘要

### 4. 安全边界

所有远程命令都应遵守以下原则：

- 默认可预览
- 高危命令必须二次确认
- 用户主密码仅用于本地加密解锁
- 私钥和密码只做本地加密存储，不上传第三方

## 状态管理说明

当前以前端原生 Hooks 为主：

- 本地状态: `useState`
- 回调缓存: `useCallback`
- 派生数据: `useMemo`
- 跨组件共享: 以 `App.tsx` 状态提升为主

这套方式在当前阶段可用，但随着功能增加，需要警惕 `App.tsx` 继续膨胀和 props drilling 加深。

## 开发约定

### TypeScript

- 优先函数式组件
- 组件 Props 在组件上方定义
- `useEffect` 必须有明确依赖
- 事件处理函数优先使用稳定引用
- 类型定义优先收口到 `src/types/index.ts`

### Rust

- 使用 `Result<T, E>` 进行错误传播
- 错误转换优先 `map_err`
- 共享状态通过 `Arc<Mutex<T>>` 管理
- 避免在锁内执行明显耗时操作

## 安全实现

- 配置加密算法: AES-256-GCM
- 密钥派生: PBKDF2
- 加密对象: 服务器密码、私钥、配置数据
- 主密码: 仅在本地会话期间驻留内存
